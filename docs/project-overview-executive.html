Executive Overview

jenkins-infra

The reusable Jenkins host for the BSF hackathon demo. This repo keeps the CI server, local bootstrap flow, and operating instructions in one place so application repos can stay focused on code, tests, and their own pipeline definitions.

Why Executives Care

This repo is the reliability layer for the demo. It turns Codex-led implementation work in hackathon-ci-demo into an auditable engineering workflow by running Jenkins locally, tracking the right repo, and validating new commits on a repeatable schedule.

Operator-facing runbooks for first-time setup, rebuild, and recovery

Strategic Separation

Why This Design Is Better

Application repos keep only code, tests, skills, and their own Jenkinsfile

Jenkins becomes reusable across multiple projects instead of being embedded in one repo

The hackathon story is cleaner: Codex builds the app, Jenkins independently validates it

Validation
Run the BSF test path

The application repo owns the actual build-and-test flow.

Jenkins runs ./ci/run_bsf_backend_tests.sh, which bootstraps Java, Maven, and MongoDB inside the app repo workspace and publishes JUnit from reports/junit/*.xml.

Why pollSCM

Designed For A Laptop Demo

Jenkins is running on localhost, so public webhooks are not required. The BSF job uses pollSCM('H/2 * * * *') to detect new commits roughly every two minutes, which keeps the setup reliable without introducing tunnel or hosting complexity.

Risk Posture

What This Repo Reduces

Reduces setup fragility by keeping Jenkins state inside the repo layout

Reduces story confusion by separating app concerns from CI host concerns

Reduces demo risk by favoring a direct local path over a heavier container dependency
