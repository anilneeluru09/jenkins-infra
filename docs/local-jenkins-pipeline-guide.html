BSF + Jenkins

Local Jenkins Pipeline Guide

This guide explains how the local Jenkins server is used with the BSF-first hackathon-ci-demo repository and how that setup supports a Codex hackathon demo on macOS.

Current Architecture

The setup is split into two repositories:

  • hackathon-ci-demo: BSF implementation, repo-local Codex skill assets, and root Jenkinsfile
  • jenkins-infra: Jenkins bootstrap scripts, runtime notes, and optional backup files

How Jenkins Starts

Start Jenkins directly on the macOS laptop with:

./scripts/start-jenkins-local.sh

That script downloads a local Java runtime, downloads the Jenkins LTS WAR, and starts Jenkins with a repo-local JENKINS_HOME.

How The BSF Repo Is Integrated

Create a Jenkins Pipeline job using:

  • Repository URL: https://github.com/anilneeluru09/hackathon-ci-demo.git
  • Branch: */main
  • Script path: Jenkinsfile

What The Pipeline Does

  1. Prevents concurrent builds.
  2. Polls SCM every two minutes with pollSCM('H/2 * * * *').
  3. Checks out the repository.
  4. Runs ./ci/run_bsf_backend_tests.sh.
  5. Bootstraps Java 21, Maven, and MongoDB inside the workspace.
  6. Runs Maven tests from BSF_NF/backend/pom.xml.
  7. Publishes JUnit results from reports/junit/*.xml.

Why This Fits The Demo

The local demo flow stays focused on BSF_NF/scripts/start-all.sh, while Jenkins uses the dedicated CI helper ./ci/run_bsf_backend_tests.sh. That keeps the macOS demo path easy to run and the CI story easy to explain.

It also supports the hackathon narrative that Codex helped with implementation, integration, and CI alignment rather than only writing code snippets.

Validation Flow

  1. Run one manual Jenkins build.
  2. Push a new commit to GitHub.
  3. Wait up to about two minutes.
  4. Confirm Jenkins starts a new build automatically.
